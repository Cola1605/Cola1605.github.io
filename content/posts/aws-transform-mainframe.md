## Three-Phase Methodology

### Architecture Overview

**Purpose**: Transform legacy mainframe applications → cloud-native microservices

### Phase 1: Reverse Engineering

**Tool**: AWS Transform for mainframe

**Process**: Extract từ existing code:
- Business logic từ COBOL
- Business rules từ Job Control Language (JCL)

**Output**: Documented business logic và rules

### Phase 2: Forward Engineering

**Generates**:

**Microservice specifications**:
- Service boundaries
- API definitions
- Communication patterns

**Modernized source code**:
- Cloud-native languages
- Modern frameworks
- Best practices applied

**Infrastructure as Code (IaC)**:
- Terraform
- AWS CDK
- CloudFormation

**Modernized databases**:
- Relational (RDS, Aurora)
- NoSQL (DynamoDB)
- Data lake solutions

### Phase 3: Deploy và Test

**Actions**:
- Deploy generated microservices using IaC
- Deploy to Amazon Web Services (AWS)
- Test modernized application functionality

**Validation**: Ensure all features working correctly

## Test Workflow: Three Planning Steps

### Purpose

**Create**: Modernization project test plan

**Generate**: Test data collection scripts

### Step 1: Configure Test Plan Input

#### Linking Capability

**Feature**: Link test plan to other job files

**Basis**: Generated dựa trên mainframe application code analysis

#### Optional Detail Enhancement

**Additional context từ**:
- Extracted business logic
- Technical documentation
- Code decomposition
- Scheduler plans

**Benefit**: More detailed test plans với additional context

### Step 2: Define Test Plan Scope

#### Entry Point Definition

**Concept**: Specific programs as entry points where application execution flows start

**Example**: Batch job JCL
- Job name
- Program sequence
- Input/output files

#### Test Case Design

**Structure**: Each functional test case designed to:
- Start execution từ specific entry point
- Follow defined execution path
- Validate expected outcomes

### Step 3: Adjust Test Plan

#### Composition

**Test plan**: Consists của test cases executed sequentially

**Order matters**: Execution sequence important

#### Test Case Details Page Capabilities

**Reorder**: Change test case sequence
- Drag and drop
- Prioritization
- Logical grouping

**Add**: New cases
- Additional scenarios
- Edge cases
- Error conditions

**Merge**: Multiple cases combined
- Consolidate similar tests
- Reduce redundancy
- Streamline execution

**Split**: One case → two
- Separate concerns
- Improve clarity
- Better isolation

#### Batch Test Cases

**Composition**: Sequence of JCL following scheduler plan
- Job dependencies
- Execution order
- Resource allocation
