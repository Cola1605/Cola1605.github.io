# Giới thiệu AWS Transform Custom: Giải quyết Tech Debt với AI-Powered Code Modernization

## Giới thiệu

Technical debt (nợ kỹ thuật) là một trong những thách thức dai dẳng nhất mà các enterprise development team phải đối mặt ngày nay.

### Thống kê Đáng lo ngại

Theo nghiên cứu, **các doanh nghiệp chi 20% ngân sách IT** - vốn dĩ nên dành cho tính năng mới - để đối phó với technical debt.

### Các Công việc Lặp đi lặp lại

**Challenges**:
- Upgrade legacy frameworks
- Migrate sang runtime versions mới
- Refactor old code patterns

**Impact**: Những công việc thiết yếu nhưng lặp đi lặp lại này tiêu tốn thời gian quý báu của developers - thời gian vốn nên dành cho innovation.

## Getting Started: Python 3.8 → 3.13 Lambda Migration

### Scenario Overview

**Context**: Python 3.8 reached End-of-Life (EOL)
- No more security updates
- Need upgrade to Python 3.13

**Demo approach**: CLI usage (Web interface cũng có powerful campaign management features)

### Step 1: List Available Transformations

#### Command

```bash
atx custom def list
```

**Alternative**: Type `atx` để access conversational interface

#### Output

**AWS-managed transformations** (identified by `AWS/` prefix):
- Managed và updated by AWS
- Production-ready
- Best practices embedded

**User-created transformations**:
- Existing custom transformations trong organization
- Shared across teams

### Step 2: Review Available Options

#### Sample AWS-Managed Transformations

**AWS/java-version-upgrade**:
- Purpose: Java runtime modernization
- Covers: Java 8 → 11 → 17 → 21

**AWS/python-boto2-to-boto3-migration**:
- Purpose: Update Python AWS SDK usage
- Migration: Boto2 → Boto3

**AWS/nodejs-version-upgrade**:
- Purpose: Node.js runtime updates
- Covers: All LTS versions

#### Selected Transformation

**For Python 3.8 → 3.13**: Use `AWS/python-version-upgrade`

### Step 3: Execute Migration

#### Command

```bash
atx custom def exec \
  -p /mnt/c/Users/vasudeve/Documents/Work/Projects/ATX/lambda/todoapilambda \
  -n AWS/python-version-upgrade \
  -C "pytest" \
  --configuration "additionalPlanContext=アップグレード先のPythonバージョンはPython 3.13" \
  -x-t
```

#### Parameters Breakdown

**-p**: Project repository path

**-n**: Transformation name (`AWS/python-version-upgrade`)

**-C**: Validation command (`pytest` để run unit tests)

**--configuration**: Specify target Python version qua `additionalPlanContext`

**-x-t**: Additional options

#### Documentation

**Full details**: Check [documentation](https://docs.aws.amazon.com/transform/latest/userguide/custom.html)

### Step 4: Migration Process

**Version-specific behavior adjustments**:

## Creating Custom Transformations

### When to Create Custom

**AWS-managed transformations**: Effectively handle common scenarios

**Custom transformations**: Needed cho organization-specific requirements

### Example: Angular 16 → 19 Migration

#### Step 1: Initialize CLI

**Command**: Type `atx` để initialize atx CLI

**First question**: Use existing transformation hoặc create new?

**Choice**: Create new

**Natural language**: All interactions qua natural language
- Input: "新しいもの"
- Alternative: "新しいものを作成したい"
- **Both understood identically**

#### Step 2: Describe Transformation

**Prompt**: What transformation do you want?

**Input**: "Angular 16から19へのアプリケーション移行"

**CLI action**: Search for all available transformations cho migration type này

**Result**: 
- Shows existing Angular migrations team đã created
- Warns: No exact match cho Angular 16 → 19
- Offers: Choose existing hoặc create custom

#### Step 3: Create Custom Transformation

**Input (natural language)**: "create a new one"

**Note**: Any expression variant works nếu intent clear

**Questions asked**:
- Documentation available?
- Sample code?
- Migration guides?
- Anything to help customize transformation plan?

#### Step 4: Provide Context

**Demo input**: "詳細はわかりません。ベストプラクティスに従ってください。"

**CLI response**: Create comprehensive transformation definition cho Angular 16 → 19 migration

**Data source**: Pre-trained data utilized để generate best practices-based results

**Recommendation**: Provide as much information và relevant data as possible cho better results

**Flexibility**: Don't need all data upfront - có thể add data throughout custom transformation definition creation process

### Step 5: Generated Transformation Definition

#### Output Format

**File**: Markup file với:
- Summary
- Comprehensive implementation steps
- Phased approach

#### Phases Included

**Phase 1: Pre-migration Preparation**
- Environment setup
- Dependency audit
- Backup creation

**Phase 2: Processing và Splitting**
- Code analysis
- Module identification
- Dependency mapping

**Phase 3: Static Dependency Analysis**
- Import resolution
- Circular dependency detection
- Version compatibility checks

**Phase 4: Specific Transformation Rule Search và Application**
- Pattern matching
- Syntax conversion
- API migration

**Phase 5: Incremental Migration và Iterative Validation**
- Step-by-step migration
- Testing after each step
- Rollback capability

#### Best Practice: Incremental Migration

**AWS Transform choice**: Thay vì direct 16 → 19
- **Staged migration**: 16 → 17 → 18 → 19
- **Reason**: Minimize issues
- **Benefit**: Each step validated before proceeding

#### Testing Phases

**Purpose**: Ensure reliable completion của each phase

**Types**:
- Unit tests
- Integration tests
- E2E tests

#### Final Verification Phase

**Included**: Comprehensive testing cho all aspects của application

**Exit criteria**: Listed requirements để accept migration as successful:
- All tests pass
- No breaking changes
- Performance maintained
- Bundle size acceptable

### Step 6: Review và Publish

#### Post-Creation Options

**Option 1**: Review hoặc modify transformation definition
- Iterate as many times as needed
- Refine until satisfactory

**Option 2**: Apply transformation definition đến Angular codebase

**Option 3**: Make available for team
- **Option 4 selected**: Publish to registry

#### Publishing to Registry

**Required fields**:
- Name
- Purpose description

**Auto-extraction**: AWS Transform automatically extracts từ context
- Suggested name: "Angular-16から19への移行"
- Clear purpose shown

**Confirmation**: "はい、良さそうです" → proceed with publishing

**Result**: Transformation definition created và published
- Can be used repeatedly
- Any code repository
- Reusable asset

### Step 7: Apply Transformation

#### Initiate Application

**Option 1 selected**: Apply transformation

**CLI prompts for**:
- Application's filesystem path
- Build command (optional)

#### Processing

**AWS Transform actions**:
- Parse codebase
- Create detailed step-by-step transformation plan
- Based on previously created definition

**Output**: JSON file với:
- Detailed migration plan
- Application-specific adjustments
- Step-by-step instructions

#### Iteration Capability

**Similar to definition creation**:
- Review plan
- Iterate as needed
- Provide feedback
- Adjust cho specific requirements

### Step 8: Execute Migration

#### Approval

**Natural language**: Tell AWS Transform để start migration process

**Example input**: "よさそうだ" → proceed

#### Execution

**Visibility**: See plan execution status trên shell

**Process**: Changes applied incrementally đến codebase

**Duration**: Varies by application complexity (demo: several minutes)

### Step 9: Completion Summary

#### Transformation Summary

**Provided information**:
- Overview của entire transformation

- Supporting evidence cho reported status

#### Example: Application Build - Production
